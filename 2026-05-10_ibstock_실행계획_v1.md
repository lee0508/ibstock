# 2026-05-10 ibstock 실행계획 v1

## 1. 문서 분석 요약

현재 프로젝트 폴더에는 기획 문서 1건만 존재한다.

- 원문서: `2026-05-10_IB수급TV 주식 정보 사이트 개발 실행 계획서.md`
- 성격: 사업 개요, 목표 아키텍처, 후보 기술, 4주 일정이 정리된 상위 기획 문서
- 현재 상태: 구현 착수 전 단계

원문서 기준으로 확인된 핵심 방향은 아래와 같다.

- 기존 `weeslee-rag` 구조를 재사용해 별도 `stock-rag` 또는 `ibstock` 서비스로 분리 구축
- 데이터 소스는 `IB수급TV` Shorts 메타데이터 및 텍스트
- 검색 방식은 FAISS + Ollama 기반 RAG
- 운영 기능은 검색 UI, 섹터 분석, IB 수급 현황, 관리자 인덱스 관리
- 자동 수집 및 인덱스 갱신까지 포함한 운영형 서비스 목표

## 2. 원문서 기준 확정 사항

아래 항목은 바로 실행 계획에 반영 가능한 확정 사항이다.

- 백엔드 기본 스택: `FastAPI`
- 벡터 검색: `FAISS`
- LLM 런타임: `Ollama`
- 기본 임베딩 모델: `nomic-embed-text`
- 기본 UI 참조: `weeslee-rag/frontend/rag-assistant.html`, `admin.html`
- 신규 핵심 엔드포인트 범위:
  - `POST /api/stock/search`
  - `GET /api/stock/sectors`
  - `GET /api/stock/ib-supply`
  - `POST /api/stock/query`
  - 관리자 인덱스 상태 및 재빌드 API
- 데이터 계층:
  - 벡터 검색용 `FAISS`
  - 메타데이터/이력용 `SQLite`
- 운영 흐름:
  - 수집
  - 전처리
  - 인덱싱
  - API 제공
  - 프론트 제공
  - 배치 자동화

## 3. 원문서의 실행상 공백

원문서는 방향은 충분하지만 실제 착수용으로는 아래가 비어 있다.

- 로컬 작업 기준 디렉터리와 실제 코드베이스 위치
- `weeslee-rag`를 복사할지, 참조 구현만 가져올지에 대한 결정
- Shorts 수집 범위:
  - 채널 1개인지 2개인지
  - Shorts 전용인지 일반 영상 포함인지
  - 전체 적재인지 최근 N일 증분 적재인지
- 종목명, 기관명, 섹터 추출 규칙의 사전 정의
- 관리자 화면에서 어떤 운영 액션을 제공할지에 대한 상세 기준
- 배포 방식:
  - 포트 기반인지 경로 기반인지
  - 기존 서버에 공존 배치할지 별도 디렉터리만 분리할지
- 검증 기준:
  - 수집 성공 기준
  - 인덱싱 성공 기준
  - 검색 정확도 샘플 기준
  - 운영 모니터링 기준

이 공백 때문에 바로 개발에 들어가면 구현보다 구조 결정과 데이터 정제가 더 오래 걸릴 가능성이 높다.

## 4. 권장 실행 전략

권장 전략은 `전체 서비스 동시 개발`이 아니라 `데이터 파이프라인 선확정 -> 검색 API 검증 -> UI 이식` 순서다.

이유는 아래와 같다.

- 이 프로젝트의 실패 지점은 프론트엔드보다 데이터 품질에 있다.
- Shorts 텍스트는 길이가 짧고 표현이 요약형이라, 종목/기관/섹터 추출 정확도가 낮으면 RAG 품질이 바로 떨어진다.
- 따라서 1차 성공 조건은 "페이지가 뜬다"가 아니라 "대표 질의에 대해 재현 가능한 답을 낸다"여야 한다.

## 5. 단계별 실행 계획

### Phase 0. 작업 기준선 확보

목표:
실제 구현을 시작할 기준 경로, 참조 자산, 산출물 구조를 먼저 고정한다.

작업:

1. `ibstock` 폴더 내 실제 코드 유무 확인
2. 참조 소스인 `weeslee-rag`의 재사용 범위 확정
3. 로컬 개발 구조 결정
4. 산출물 디렉터리 초안 생성

산출물:

- 프로젝트 초기 디렉터리 구조
- 참조 자산 매핑 메모
- 환경 변수 목록 초안

완료 기준:

- 어떤 파일을 신규 생성하고 어떤 파일을 복사/참조할지 정해짐
- 구현 시작 경로가 고정됨

### Phase 1. 데이터 계약 확정

목표:
수집 데이터가 이후 DB, FAISS, API, UI에서 동일하게 쓰이도록 메타데이터 스키마를 먼저 확정한다.

작업:

1. Shorts 원본 JSON 샘플 30~50건 수집
2. 아래 표준 필드 확정
3. 분류 사전 초안 작성
4. 적재 정책 확정

필수 필드:

- `video_id`
- `channel`
- `title`
- `published_at`
- `views`
- `raw_text`
- `stocks`
- `ib_names`
- `sectors`
- `category`
- `source_url`
- `collected_at`

사전 정의가 필요한 규칙:

- 종목명 표준화 규칙
- 기관명 표준화 규칙
- 섹터 다중 매핑 허용 여부
- `buy/watch/sell` 분류 기준

산출물:

- 샘플 원본 JSON
- 정규화 스키마 문서
- 키워드/사전 파일 초안

완료 기준:

- 동일 영상이 어떤 구조로 DB와 FAISS에 저장되는지 예시 3건 이상 확정

### Phase 2. 수집 및 전처리 파이프라인

목표:
반복 실행 가능한 수집-정제-적재 파이프라인을 먼저 만든다.

작업:

1. `collect_shorts.py` 작성
2. `preprocess.py` 작성
3. `sqlite load` 단계 작성
4. 실패 재실행 가능한 로그 구조 추가

구현 우선순위:

1. 최근 30건 수집
2. 단일 실행으로 JSON 저장
3. 정규화 후 SQLite 적재
4. 중복 제거
5. 증분 수집

산출물:

- `data/raw/*.json`
- `data/processed/*.json`
- `data/db/*.db`
- 실행 로그

완료 기준:

- 한 번의 명령으로 최근 샘플 데이터를 수집하고 DB에 적재 가능
- 중복 실행 시 중복 레코드가 폭증하지 않음

### Phase 3. 검색 인덱스 구축

목표:
텍스트 품질과 검색 결과를 먼저 검증할 최소 인덱스를 만든다.

작업:

1. 청킹 정책 결정
2. 임베딩 생성
3. FAISS 인덱스 빌드
4. 메타데이터 매핑 저장
5. 샘플 질의 검색 테스트

권장 방식:

- 초기에는 영상 30~100건만 사용
- 청크는 영상 단위 또는 짧은 분할 단위로 단순하게 시작
- 먼저 검색 재현성을 확인한 뒤 고급 전처리 추가

검증 질문 예시:

- `골드만삭스가 최근 언급한 종목은?`
- `전력 인프라 관련 종목만 보여줘`
- `JP모건과 함께 언급된 반도체 종목은?`

산출물:

- `faiss_index/`
- 검색 테스트 결과 메모

완료 기준:

- 대표 질의 5개 중 4개 이상에서 관련 문서가 상위 결과에 노출

### Phase 4. 백엔드 API 구성

목표:
프론트보다 먼저 API 계층을 고정해 UI 변경 비용을 줄인다.

작업:

1. FastAPI 앱 초기 구조 생성
2. SQLite 조회 서비스 작성
3. FAISS 검색 서비스 작성
4. Ollama 질의 응답 서비스 연결
5. 관리자용 상태 API 연결

우선 구현 API:

- `GET /api/health`
- `GET /api/stock/sectors`
- `GET /api/stock/ib-supply`
- `POST /api/stock/search`
- `POST /api/stock/query`

산출물:

- 실행 가능한 API 서버
- 샘플 요청/응답 문서

완료 기준:

- curl 또는 브라우저 호출로 핵심 API 응답 확인 가능
- 검색과 질의응답이 동일 메타데이터 구조를 사용

### Phase 5. 프론트엔드 이식

목표:
기존 `weeslee-rag` UI를 완전 재작성하지 말고, 구조를 활용해 주식 전용 화면으로 빠르게 전환한다.

작업:

1. 기존 `rag-assistant.html` 구조 분석
2. 공통 레이아웃과 API 호출부 재사용
3. 아래 4개 화면 우선 구현

우선 화면:

- `index.html`
- `stock-search.html`
- `sector.html`
- `ib-supply.html`

후순위:

- `admin.html` 주식 도메인 맞춤화

완료 기준:

- 종목 검색, 섹터 보기, 기관 수급 조회가 각각 API와 연결됨
- 더미 데이터가 아니라 실제 SQLite/FAISS 결과가 출력됨

### Phase 6. 운영 자동화 및 배포

목표:
로컬 동작 확인 뒤 자동 수집과 서버 배포를 마지막에 고정한다.

작업:

1. 스케줄러 작성
2. 증분 수집 실행 경로 점검
3. 서비스 실행 명령 정리
4. Nginx 경로 또는 포트 배치
5. 운영 검증

운영 검증 항목:

- 새 Shorts 수집 성공
- DB 적재 성공
- 인덱스 재생성 성공
- API 정상 응답
- 프론트 화면 정상 호출

완료 기준:

- 수동 재빌드 없이 일일 수집과 검색 반영 흐름이 재현됨

## 6. 실제 착수 순서

가장 합리적인 실제 작업 순서는 아래다.

1. `weeslee-rag` 참조 구조 확인
2. `ibstock` 프로젝트 골격 생성
3. Shorts 수집 샘플 스크립트 작성
4. SQLite 스키마 및 적재 구현
5. FAISS 샘플 인덱스 구축
6. 검색 API 구현
7. 질의응답 API 구현
8. 프론트 1차 연결
9. 자동화/배포

## 7. 주요 리스크

### 데이터 리스크

- Shorts 텍스트가 짧아 종목/기관/행위 추출 정확도가 낮을 수 있음
- 제목만으로는 `매수`, `매집`, `순매수` 의미가 불명확할 수 있음

대응:

- 초기에는 규칙 기반 추출로 시작
- 불확실 항목은 `unknown` 또는 복수 후보 허용
- 대표 영상 수동 검수셋을 따로 둠

### 시스템 리스크

- `weeslee-rag` 재사용 범위가 불명확하면 복사 비용이 커질 수 있음
- Ollama 모델 응답 속도가 실서비스 UX를 저해할 수 있음

대응:

- 먼저 최소 기능만 복제
- RAG 검색과 생성형 응답을 분리 검증

### 운영 리스크

- YouTube 수집 방식이 바뀌면 `yt-dlp` 의존성이 흔들릴 수 있음
- 전체 재인덱싱이 매일 실행되면 운영 비용이 커질 수 있음

대응:

- 증분 적재 기준 설계
- 전체 재빌드는 관리자 수동 기능으로 남김

## 8. 1차 구현 범위 권장안

첫 구현은 아래 범위로 제한하는 것이 적절하다.

- 채널 1개
- 최근 30~100개 Shorts
- 규칙 기반 종목/기관/섹터 추출
- SQLite + FAISS 최소 동작
- 검색 UI 1개 + 섹터/수급 조회 화면 2개

아래는 2차로 미루는 것이 합리적이다.

- 다중 채널 통합
- 복잡한 한국어 NER 고도화
- 고급 관리자 통계
- 완전 자동 운영 배포

## 9. 다음 실행 권고

현재 폴더 상태 기준으로는 코드보다 문서만 있으므로, 다음 작업은 계획 수립이 아니라 바로 `Phase 0~2` 착수다.

즉시 시작 우선순위:

1. 참조 코드베이스 위치 확인
2. 프로젝트 초기 디렉터리 생성
3. Shorts 샘플 수집 스크립트 작성
4. SQLite 스키마 생성

이 4개가 끝나야 이후 API와 UI 작업이 의미를 가진다.
